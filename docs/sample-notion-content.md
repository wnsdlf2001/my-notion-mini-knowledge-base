### 문서 1 — 본문 (이 아래 줄부터 복사)

## 훅 규칙 두 가지

React 훅에는 반드시 지켜야 하는 두 가지 규칙이 있습니다.

## 1. 최상위에서만 호출

반복문, 조건문, 중첩 함수 안에서 훅을 호출하면 안 됩니다. 컴포넌트가 렌더링될 때마다 훅이 동일한 순서로 호출되어야 React가 상태를 올바르게 추적할 수 있기 때문입니다.

> 훅 호출 순서가 렌더링마다 달라지면 상태가 엉킵니다.

## 2. React 함수 안에서만 호출

훅은 React 컴포넌트 또는 커스텀 훅 내부에서만 호출합니다.

(여기서 한 줄 띄우고 `/코드` 로 코드블록을 만들어 아래를 넣으면 코드블록 렌더링도 검증됩니다)

const Example = () => {
  const [count, setCount] = useState(0)
  return count
}

### 문서 2 — 본문 (이 아래 줄부터 복사)

## Hydration이란

서버에서 렌더링한 HTML에 클라이언트 JavaScript가 이벤트를 붙이는 과정입니다.

## 에러 원인

서버 렌더 결과와 클라이언트 첫 렌더 결과가 다르면 발생합니다.

- 브라우저 전용 API(window, localStorage)를 렌더 중 사용
- new Date() 같은 매 렌더 달라지는 값
- 잘못된 HTML 중첩 구조

## 해결 방법

useEffect 안에서 처리하거나, 마운트 이후에만 렌더하도록 분기합니다.

> next-themes를 쓸 때 html 태그에 suppressHydrationWarning을 넣는 이유도 이것입니다.

### 문서 3 — 본문 (이 아래 줄부터 복사)

## 핵심 차이

REST는 여러 엔드포인트로 자원을 나누고, GraphQL은 단일 엔드포인트에서 쿼리로 필요한 데이터만 요청합니다.

## REST 특징

- 자원마다 URL이 분리됨
- 오버페칭/언더페칭이 발생하기 쉬움
- HTTP 캐싱 활용이 쉬움

## GraphQL 특징

- 필요한 필드만 정확히 요청
- 단일 엔드포인트
- 스키마 기반 타입 시스템

## 언제 무엇을

단순 CRUD에는 REST, 복잡하고 다양한 데이터 조합이 필요하면 GraphQL이 유리합니다.

### 문서 4 — 본문 (이 아래 줄부터 복사)

## 프로세스

실행 중인 프로그램의 인스턴스입니다. 각자 독립된 메모리 공간을 가집니다.

## 스레드

프로세스 안에서 실행되는 흐름의 단위입니다. 같은 프로세스의 스레드들은 메모리를 공유합니다.

## 비교

- 프로세스: 격리됨, 생성 비용 큼, 통신(IPC) 비용 큼
- 스레드: 메모리 공유, 생성 비용 작음, 동기화 문제 주의

> 공유 자원에 동시에 접근하면 race condition이 생길 수 있습니다.
